# Research Notes

These notes record the assumptions behind the initial Proof of Pulse POC plan. They should be updated whenever implementation discovers different platform behavior.

## Current Findings

- HealthKit is a user-permissioned store for health and fitness data on iPhone and Apple Watch. Apps can read and write supported data types when authorized.
- HealthKit requires fine-grained authorization by data type. The app should request only the data needed for the current proof.
- HealthKit intentionally prevents an app from always distinguishing denied read permission from absent data. Users may also grant only a limited recent history window.
- HealthKit samples include source-related information after they are saved and later retrieved from the HealthKit store.
- `HKObject` exposes source-related fields such as source, source revision, device, and metadata for retrieved samples.
- `HKSourceRevision` describes the source of a HealthKit sample, including version, operating system, and product type information.
- `HKDevice` can describe devices such as Apple Watch, iPhone, or health peripherals that generated stored samples.
- `HKMetadataKeyWasUserEntered` indicates whether a sample was entered by the user, when the key is present and correctly set.
- HealthKit device fields can include arbitrary strings, and app-created samples can supply device and metadata values.
- HealthKit supports a digital-signature metadata path for some tamper-resistant measuring devices with known public keys, but this should be treated as future research unless real Apple Watch samples expose verifiable signatures.
- HealthKit samples are not equivalent to sensor-level attestations. A sample can come from multiple sources, and apps must treat source metadata as a trust signal, not proof.
- App Attest can help a backend verify that requests come from a legitimate app instance and can bind requests to server challenges.
- App Attest is not available on every device, so V0 needs an explicit unsupported-device path.
- App Attest does not prove that HealthKit data was generated by Apple Watch sensors or by the same human using the account.
- DeviceCheck can provide fraud-friction signals such as repeat-device state, but it is not a proof of identity or personhood.
- Apple treats health and fitness data as especially sensitive. The app should not use HealthKit-derived data for advertising, marketing, or data mining, and should avoid raw health-data transmission.
- Apps using HealthKit need HealthKit usage descriptions and a privacy policy.
- App Store review risk is material if the app's main value proposition is a generic proof-of-human gate rather than a clear health or fitness purpose.

## Practical POC Implications

- Do feature extraction on-device.
- Request read permissions only for V0.
- Send only coarse proof claims and score tiers to the backend.
- Filter out samples marked user-entered.
- Prefer recent, multi-signal, Apple Watch-like sources.
- Reject third-party-written samples for high-confidence V0 scoring.
- Treat missing data as insufficient signal, not evidence that data does not exist.
- Require challenge freshness and replay protection.
- Use App Attest for app integrity, while documenting its limits.
- Consider DeviceCheck only for rate limiting and repeated-device fraud flags.
- Avoid sensitive HealthKit categories that are unnecessary for liveness.
- Keep the first proof short-lived to reduce tracking and replay value.
- Prefer score tiers and confidence labels over exact numeric scores in public or verifier-facing outputs.
- Keep abuse monitoring detached from raw health data.
